# Dokumentationsaudit und Bereinigungsplan für H-Town Combat 67

## Executive Summary

Ich konnte den bereitgestellten Google-Drive-Ordner über den aktivierten Connector inventarisieren; es liegen dort **53 Markdown-Dateien** vor. Der bereitgestellte GitHub-Repo-Link ließ sich mit den hier verfügbaren Mitteln **nicht direkt als vollständiger Dateibaum enumerieren**. Deshalb ist die folgende Repo-Pfad-Spalte dort, wo möglich, **aus internen Relativlinks der Dokumente erschlossen**; wo das nicht belastbar möglich war, ist der Pfad als **unspecified** markiert. Die Connector-Ergebnisse zeigen außerdem nur **Google Drive** als nutzbaren verbundenen Dokumenten-Connector in diesem Auftrag. 

Die Dokumentation ist inhaltlich **stark und tief**: Es gibt bereits einen klaren **kanonischen Kern** aus `20H_DOCUMENTATION_INDEX`, `20H_AREA_PROGRESSION_MODEL`, `Technical_Architecture`, `COMPLETE_GAME_FUNCTION_SPEC` sowie den 20H-Content-Verträgen für Systeme, Content-Schema, Quests, Regionen, Status, Multiplayer und Lokalisierung. Dieser Kern ist für Implementierung und Review grundsätzlich tragfähig. fileciteturn1file1 fileciteturn2file1 fileciteturn8file0

Das Hauptproblem ist nicht fehlende Tiefe, sondern **mangelnde Kuratierung**: Aktive, aktuelle Spezifikationen liegen im selben Sichtfeld wie alte Unity-/Football-/REDLINE-FC-Artefakte, mehrere Status- und Roadmap-Dateien widersprechen einander, technische Aussagen driften auseinander, und die Dokumentationsoberfläche ist für neue Bearbeiter unnötig schwer lesbar. Besonders problematisch sind der Konflikt zwischen **30 Sekunden vs. 3,0 Sekunden Extraktionshaltezeit**, der Konflikt **React 19 vs. React 18**, die Vermischung von **Next.js-App-Router-Architektur mit Vite-Build-Aussagen**, sowie der fortbestehende Namens- und Scope-Leak von **REDLINE FC / Arcade Football / Unity** in aktiven oder halbaktiven Dateien. fileciteturn8file0 fileciteturn10file0 fileciteturn6file1 fileciteturn2file1 fileciteturn8file1 fileciteturn1file0 fileciteturn9file1 fileciteturn9file2

Die empfohlene Bereinigung ist klar: **kanonische Spine behalten**, **Status/QA/Operations konsolidieren**, **Legacy hart archivieren**, **Dateinamen/Front Matter standardisieren**, **automatisierte Checks verpflichtend machen**, und **eine kleine Zahl neuer “Owner”-Dokumente** einführen, damit künftig nicht wieder parallele Wahrheiten entstehen. Die wichtigste organisatorische Änderung ist: `README.md` wird wieder zu einer **schlanken Einstiegseite**, während `docs/project/status.md`, `docs/project/roadmap.md`, `docs/qa/*` und `docs/deployment/*` die operative Last übernehmen. Diese Richtung passt auch zur bestehenden Source-of-Truth-Hierarchie im Dokumentationsindex und schließt die vom eigenen Audit bereits benannten Restlücken systematisch. fileciteturn1file1 fileciteturn2file2

## Qualitätsmängel und konkrete Umschreibungen

### Einstiegsdokument entlasten

`README.md` ist aktuell gleichzeitig Projektpitch, Architekturindex, Statusübersicht, Implementierungsleitfaden, Entwicklungsstatus, Checkliste und Next-Step-Liste. Das macht die Datei nützlich für einmaliges Lesen, aber schlecht als langfristig stabilen Einstiegspunkt. Gleichzeitig sagt das Vollständigkeits-Audit selbst, dass `README.md` zu hochlevelig für exakte Verhaltensdefinitionen ist. fileciteturn0file0 fileciteturn2file2

**Vorher, sinngemäß:** großer Überblick + viele Deep-Links + lange Statuscheckliste in derselben Datei. fileciteturn0file0

**Nachher, vorgeschlagen:**
```md
# H-Town Combat 67

Browser-native vehicle extraction roguelite with a 20h campaign documentation pack.

## Start here
1. docs/project/status.md
2. docs/preproduction/20H_DOCUMENTATION_INDEX.md
3. docs/Technical_Architecture.md
4. docs/COMPLETE_GAME_FUNCTION_SPEC.md

## Language policy
- Technical docs: English
- Player-facing text: DE/EN localization files

## Where to find what
- docs/project/     status, roadmap, versioning
- docs/preproduction/ content contracts
- docs/qa/          release and test evidence
- docs/archive/     historical documents
```

**Effekt:** geringere Driftfläche, schnellere Orientierung, weniger Status-Duplikation.

### Status und Roadmap trennen

Heute verteilen sich aktuelle, historische und geplante Zustände auf `README.md`, `MVP_Scope.md`, `ROADMAP.md`, `ROADMAP_NEXT.md`, `ROADMAP_DETAILED.md`, `PHASE_1_STATUS.md` und teils `repo-status.md`. Mindestens ein harter Konflikt ist bereits sichtbar: “Phase 2 in progress” vs. “Phase 2 sprint complete.” fileciteturn0file0 fileciteturn9file0

**Vorher, sinngemäß:** mehrere Dateien führen gleichzeitig “aktuellen Stand” und “nächste Schritte”. fileciteturn0file0 fileciteturn9file0

### Historische Dokumente mit harter Archiv-Bannerlogik versehen

Mehrere Dokumente haben bereits einen “historical note”, bleiben aber trotzdem im aktiven Dateibaum und sehen auf den ersten Blick wie normale Projektunterlagen aus. Das gilt besonders für `PreProduction_MasterDoc.md`, `Coding_Conventions.md` und das `ULTIMATE_H_TOWN_MASTER_MANUAL.md`. fileciteturn9file1 fileciteturn8file2 fileciteturn10file0

### Extraktionsregel kanonisieren

Der Extraktionskonflikt ist zu zentral, um “nebenbei” weiterzuleben. Die kanonische Datei (`COMPLETE_GAME_FUNCTION_SPEC.md`) sollte den Default explizit festlegen, und alle historischen Abweichungen müssen klar als historisch oder experimentell markiert werden. fileciteturn8file0 fileciteturn10file0 fileciteturn6file1

**Vorher, sinngemäß:** ein Dokument spricht von 30 Sekunden Default, ein anderes von 3,0 Sekunden Implementierungswert. fileciteturn8file0 fileciteturn10file0

**Nachher, vorgeschlagen:**
```md
## Extraction Hold Rule
Default extraction hold time is 30 seconds.

If implementation experiments use a different value, they must be documented as:
- non-default
- branch-specific
- temporary
- not canonical until adopted here
```

**Effekt:** QA, Design und Implementierung bekommen dieselbe Basisregel.

Die offiziellen Next.js-Dokumente beschreiben für aktuelle Projekte `next dev`, `next build`, `next start` und `eslint`, nicht einen Vite-zentrierten Standardweg. React 19 ist stabil; TypeScript 6 ist offiziell veröffentlicht. fileciteturn1file0 fileciteturn2file1 fileciteturn8file1 citeturn8search0turn9search0turn7search0

**Vorher, sinngemäß:** Next.js + React + Three.js + Zustand + Socket.io, aber zusätzlich Vite als Build Tool und React 18 in anderer Datei. fileciteturn1file0 fileciteturn8file1

## Zielstruktur, Versionierung und Bereinigung

### Empfohlene Kanonisierung

Die vorhandene Dokumentation sollte nicht platt “verschlankt”, sondern **in eine stabile Navigationsarchitektur überführt** werden. Die folgende Zuordnung minimiert Drift und respektiert zugleich bestehende, bereits gute Dokumente.

## Automatisierung, Skripte und reproduzierbare Deliverables

Die Bereinigung sollte nicht manuell “nebenbei” passieren, sondern durch feste Tooling-Gates abgesichert werden. Dafür eignen sich vier Werkzeuge sehr gut: **markdownlint** für Markdown-Stil und Struktur, **Vale** für Sprachstil und Terminologie, **lychee** für Link-Prüfung, und **pre-commit** für lokale Hook-Integration. markdownlint beschreibt sich selbst als Linter für Markdown/CommonMark und unterstützt GFM-Syntax; Vale ist für stilistische Konsistenz und projektspezifische Vokabularregeln gedacht; lychee ist ein asynchroner Linkchecker mit rekursiver Prüfung und Offline-Modus; pre-commit verwaltet sprachübergreifende Hooks und kann auf allen Dateien laufen. citeturn12search1turn10search8turn11search7turn11search4turn14search1turn14search3turn10search4

Für das JavaScript-/Next-Tooling ist außerdem wichtig, die aktuellere Next.js-Konvention zu verwenden: Die offiziellen Next.js-Dokumente zeigen `next dev`, `next build`, `next start` und `eslint` in `package.json`; wenn noch alte `next lint`-Annahmen im Umlauf sind, sollten diese bereinigt werden. citeturn8search0

### Empfohlene Kommandos

```bash
# Markdown-Struktur prüfen
npx markdownlint-cli2 "README.md" "docs/**/*.md"

# Prosa- und Terminologieprüfung
vale README.md docs

# Linkprüfung
lychee --config lychee.toml README.md docs/**/*.md

# Alle lokalen Hooks auf gesamten Bestand ausführen
pre-commit run --all-files
```

### Beispiel für Hook-Konfiguration

```yaml
repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v5.0.0
    hooks:
      - id: end-of-file-fixer
      - id: trailing-whitespace
      - id: mixed-line-ending
      - id: check-yaml
      - id: check-json

  - repo: https://github.com/DavidAnson/markdownlint-cli2
    rev: v0.18.1
    hooks:
      - id: markdownlint-cli2
        files: \.md$

  - repo: local
    hooks:
      - id: vale
        name: vale
        entry: vale
        language: system
        types: [markdown]
      - id: lychee
        name: lychee
        entry: bash -lc 'lychee --config lychee.toml README.md docs/**/*.md'
        language: system
        pass_filenames: false
```

**Wirkung:** Jeder Commit scheitert früh an kaputten Links, Terminologielecks, Stilproblemen oder whitespace-/EOF-Themen, statt diese erst im Review zu entdecken. pre-commit ist genau für solche sprachübergreifenden Hook-Setups gedacht. citeturn10search4

### Beispiel für eine einfache Python-Audit-Datei

```python
#!/usr/bin/env python3
from pathlib import Path
import hashlib
import re

LEGACY_TERMS = [
    r"REDLINE FC",
    r"Arcade Football",
    r"MonoBehaviour",
    r"Unity 6",
    r"The Autoballer",
]

def sha1(text: str) -> str:
    return hashlib.sha1(text.encode("utf-8")).hexdigest()

def files():
    for p in Path(".").rglob("*.md"):
        if ".git" not in p.parts:
            yield p

seen = {}
for p in files():
    text = p.read_text(encoding="utf-8", errors="ignore")
    digest = sha1(text)
    print(f"[FILE] {p}")
    if digest in seen:
        print(f"  duplicate-of: {seen[digest]}")
    else:
        seen[digest] = p
    for pattern in LEGACY_TERMS:
        for m in re.finditer(pattern, text, re.IGNORECASE):
            line = text[:m.start()].count("\\n") + 1
            print(f"  legacy-hit: {pattern} @ line {line}")
```

**Wirkung:** erkennt 1:1-Duplikate und Legacy-Begriffe früh, bevor sie wieder in kanonische Dateien zurückdiffundieren.

### Reproduzierbare Clean-up- und ZIP-Schritte

Ich konnte in diesem Zugriffskontext **keine Dateien schreiben**, **keine Quellablage ändern** und **kein ZIP-Artefakt erzeugen**. Deshalb liefere ich hier den exakten reproduzierbaren Ablauf. Er ist absichtlich robust gegen unklare alte Pfade und sucht zuerst im Root, dann unter `docs/`.

```bash
#!/usr/bin/env bash
set -euo pipefail

branch="docs/cleanup-2026-04-26"
git checkout -b "$branch"

mkdir -p docs/project docs/qa docs/deployment docs/adr docs/assets docs/archive/legacy scripts

move_any() {
  local name="$1"
  local target="$2"
  mkdir -p "$(dirname "$target")"
  if [ -f "$name" ]; then
    git mv "$name" "$target"
    return 0
  fi
  if [ -f "docs/$name" ]; then
    git mv "docs/$name" "$target"
    return 0
  fi
  echo "WARN missing: $name" >&2
}

# Legacy archive
move_any "MASTER_DOCUMENTATION.md" "docs/archive/legacy/master-documentation.md"
move_any "PreProduction_MasterDoc.md" "docs/archive/legacy/preproduction-masterdoc.md"
move_any "Complete_Game_Development_Checklist.md" "docs/archive/legacy/complete-game-development-checklist.md"
move_any "Coding_Conventions.md" "docs/archive/legacy/coding-conventions-unity.md"
move_any "ULTIMATE_H_TOWN_MASTER_MANUAL.md" "docs/archive/legacy/ultimate-h-town-master-manual.md"
move_any "ROADMAP_NEXT.md" "docs/archive/legacy/roadmap-next.md"
move_any "Repo_Setup_Status.md" "docs/archive/legacy/repo-setup-status.md"
move_any "GDD_Masterplan_ArcadeFootball.md" "docs/archive/legacy/gdd-masterplan-arcade-football.md"
move_any "Project_Settings_Policy.md" "docs/archive/legacy/project-settings-policy.md"
move_any "Unity_Script_Scaffold_Index.md" "docs/archive/legacy/unity-script-scaffold-index.md"

# Create canonical project docs
cat > docs/project/status.md <<'EOF'
# Project Status
- Phase 1 complete
- Canonical docs spine exists
- Browser/device QA still open
- Performance p99 spikes still under investigation
EOF

cat > docs/project/roadmap.md <<'EOF'
# Project Roadmap
- Finish documentation cleanup
- Close browser/device validation
- Investigate performance spikes
- Complete content implementation
EOF

cat > docs/project/versioning.md <<'EOF'
# Documentation Versioning Policy
Use front matter with: title, owner, status, version, last_reviewed.
EOF

cat > docs/deployment/deployment.md <<'EOF'
# Deployment
Manual deploy only. No auto-pull and no automatic GitHub deploy workflow.
EOF

cat > docs/adr/0001-unity-to-web.md <<'EOF'
# Move from Unity to Web Stack
Accepted. Canonical architecture lives in Technical_Architecture.md.
EOF

# Optional asset merge stub
cat > docs/assets/asset-pipeline.md <<'EOF'
# Asset Pipeline
Merged replacement for ASSET_MANIFEST.md and ASSET_GENERATION_REPORT.md.
EOF

# Tooling config
cat > .markdownlint.jsonc <<'EOF'
{
  "default": true,
  "MD013": false,
  "MD033": false,
  "MD041": false
}
EOF

cat > lychee.toml <<'EOF'
max_concurrency = 8
timeout = 20
retries = 2
exclude_path = ["docs/archive/**", "tmp/**", "node_modules/**"]
EOF

# Run checks
npx markdownlint-cli2 "README.md" "docs/**/*.md"
vale README.md docs
lychee --config lychee.toml README.md docs/**/*.md
pre-commit run --all-files || true

# Pack artifact
zipname="htown-docs-cleanup-$(date +%F).zip"
zip -r "$zipname" README.md docs .markdownlint.jsonc lychee.toml scripts .pre-commit-config.yaml .vale.ini
echo "Created: $zipname"
```
